# User Story A.2: Create StructuredTextSplitter

## Goal Description
Implement a smarter chunking strategy for the Markdown output generated by Docling. Instead of arbitrary character splitting, `StructuredTextSplitter` will:
1.  **Split by Headers**: Respect the document outline (H1, H2, H3).
2.  **Preserve Semantic Units**: Keep Tables and Images (and their captions) as single chunks given they are distinct markdown elements.
3.  **Metadata**: Ensure every chunk retains its source metadata (page number, original filename).

## Proposed Architecture
We will utilize `LangChain`'s `MarkdownHeaderTextSplitter` as the foundation.

- **Input**: `List[Document]` (where `page_content` is the full MD of the PDF).
- **Process**:
    1.  Define headers to split on (`#`, `##`, `###`).
    2.  Split content.
    3.  Merge metadata: The `MarkdownHeaderTextSplitter` moves headers into metadata. We need to **merge** this with the original document metadata (source, page, file_path).
- **Output**: `List[Document]` (chunks).

## Detailed Design
### Class: `StructuredTextSplitter`
- **Inheritance**: Can implement `BaseDocumentTransformer` from LangChain or just be a standalone class for now (MVP).
- **Methods**:
    - `__init__(headers_to_split_on: List[Tuple[str, str]])`
        - Default: `[("#", "Header 1"), ("##", "Header 2"), ("###", "Header 3")]`
    - `split_text(text: str) -> List[Document]`
    - `split_documents(documents: List[Document]) -> List[Document]`

### Edge Cases
- **Tables**: Markdown tables are just text. If a table is inside a section, it stays in that section chunk. If the section is too large, we might need a secondary `RecursiveCharacterTextSplitter`.
    - *Plan*: For this story (A.2), we stick to Header splitting. We assume Docling's markdown structure is good enough.
- **Missing Headers**: Content without headers will be one large chunk.

## API Changes
No external API changes. This is an internal utility used by the `IngestionPipeline` (in future steps).
